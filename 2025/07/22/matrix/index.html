1024方阵转置与底层cache原理

进行矩阵的测试的时候，会发现一个神奇的现象。1024x1024方阵的原地转置，竟然比1025x1025方阵转置更慢！这其实是一个比较反直觉的现象，其底层原理和cache是密不可分的。下面从理论和实践上分析了产生这一现象的深层次原因。

现象

进行矩阵的测试的时候，会发现一个神奇的现象。$10241024$ 方阵的原地转置，竟然比$10251025$方阵转置更慢！

当然，这里有一个非常重要的前提条件是，矩阵的元素在空间中连续！也就是，一维数组表示二维矩阵。如果是不连续的话，是没有这种现象的。

分析

很明显，这个性能差异与算法无关。代码一致，差异只能来自于cache上了。

这个含义是：

L1 Data 48KB 64Byte 12路组相连

那总的组数也很容易计算了：
$48 * 1024 / (64 * 12) = 64$组

根据计组的知识，可以清楚的发现，内存地址的后6位，作为了块内的索引。
后6-12位，做了组的索引。

同理，可以获取L2的大小信息

这个含义是：

层级 总容量 cache line 路数
L2 2048KB 64Byte 16路组相连

那总的组数也很容易计算了：
$2048 * 1024 / (64 * 16) = 2048$组

基础知识

这里是csapp里的cache示意图。如果不了解cache的原理的话，就回头去温习下。这里不做更多的讲解了。

只需要记住：对于我的L1 cache来讲：

  • 内存地址的倒数11-6位的大小，就是被分为到了cache的第几个个组。
  • 每个组有12路，当12路都被占用后，组里再来一个新的，就需要挑一个旧的cache line给扔掉。

对于我的L2 cache来讲：

  • 内存地址的倒数16-6位的大小，就是被分为到了cache的第几个个组。
  • 每个组有16路，当16路都被占用后，组里再来一个新的，就需要挑一个旧的cache line给扔掉。

在思考原因的过程中，一开始，我的思路也是一直在L1 cache，怎么都想不明白，为什么会有性能上的差异。

当思路来到L2 cache之后，情况就不一样了。

介绍完了前面的基础知识，那就可以分析原因了

valgrind验证猜想

这里我之所以采用valgrind, 而不是perf。就两个原因：

  • wsl中，perf的安装需要从源码github
    编译，但费劲编译后，还不具备硬件级采样的功能，比较困难完成cache miss率的统计
  • valgrind通过模拟cpu，结果绝对的可信

运行过程中，可能会出现warning: L3 cache found, using its data for the LL simulation.。无需在意，从生成的cachegrid.output.{pid}文件中，可以看到，模拟的cache大小是完全正确的。

用cg_annotate cachegrind.out.{pid}命令，可以提供详细的报告。
